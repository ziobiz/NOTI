# ChillPay 무효/환불 시 가맹점·전산 노티 전송 구조 정리

## 질문 요약

- API로 ChillPay 무효/환불 **완료**를 알게 된 뒤 노티를 보내는지?
- 무효/환불 **요청 시점**에 노티를 보내는지, **요청 후 API로 확인된 뒤** 노티를 보내는지?

---

## 현재 구현 (코드 기준)

### 1. 우리가 무효/환불을 **요청**하는 경우 (버튼 클릭)

| 구분 | 흐름 | 노티 시점 |
|------|------|-----------|
| **무효 요청** | `chillPayRequestVoid(txId)` 호출 → **API 성공 응답** → `sendVoidOrRefundNoti(log, 'void')` 호출 | **요청 API가 성공하면 곧바로** 가맹점·전산에 무효 노티 전송 |
| **환불 요청** | `chillPayRequestRefund(txId)` 호출 → **API 성공 응답** → `sendVoidOrRefundNoti(log, 'refund')` 호출 | **요청 API가 성공하면 곧바로** 가맹점·전산에 환불 노티 전송 |

- **구조**: **요청 성공 시점에 바로 노티** (별도의 “완료 조회 API” 후 노티 아님).
- 적용 위치:
  - 피지 노티 `/admin/logs` 의 무효 요청 / 환불 요청
  - 취소환불 메뉴 `/admin/cancel-refund/void` 의 무효 요청, `/admin/cancel-refund/refund` 의 환불 요청

---

### 2. ChillPay에서 **이미 무효 처리된 건** (수동 무효 등)

| 구분 | 흐름 | 노티 시점 |
|------|------|-----------|
| **무효 동기화** | `chillPaySearchVoid()` 로 **무효 처리된 건 목록 조회** → 우리 로그와 매칭 → 미전송 건만 `sendVoidOrRefundNoti(log, 'void')` | **API로 무효 완료를 확인한 뒤** 해당 건에 대해 노티 전송 |

- **구조**: **API로 “무효 완료”를 확인한 뒤 노티** (Search Void → 매칭 → 미전송분만 노티).
- 환불: **Search Refund 기반 동기화는 현재 없음** (Search Void만 구현됨).

---

## 정리

| 시나리오 | 노티 전송 시점 | 비고 |
|----------|----------------|------|
| 우리가 **무효 요청** 버튼 클릭 | Request Void API **성공 응답 직후** 노티 | “완료 조회” 없이 요청 성공만 보고 노티 |
| 우리가 **환불 요청** 버튼 클릭 | Request Refund API **성공 응답 직후** 노티 | 동일 |
| ChillPay **수동 무효** 등 | Search Void API로 **무효 건 조회 후** 매칭·미전송분만 노티 | “API로 완료 확인 후 노티” 구조 |
| ChillPay **수동 환불** 등 | **구현됨** | Search Refund API로 환불 건 조회 후 매칭·미전송분만 환불 노티 (취소환불 > 환불거래 **"ChillPay 환불 건 동기화"** 버튼). |

---

## 결론

- **요청 시**:  
  “무효/환불 **요청 API를 보내고, 그 API가 성공하면** 가맹점·전산에 노티를 보내는” 구조입니다.  
  **요청 후, Get Payment Details 등으로 ‘완료’를 따로 조회한 뒤 노티**하는 구조는 **아닙니다**.

- **이미 처리된 건(수동 무효)**:  
  Search Void API로 “무효 완료된 건”을 조회한 뒤, 그 결과를 기준으로 노티를 보내는 구조입니다.  
  환불은 동일한 “API로 완료 확인 후 노티” 방식의 Search Refund 동기화는 **아직 없습니다**.

원하시면 “요청 → (별도 API로 완료 확인) → 노티” 방식으로 바꾸는 설계/구현 방안도 정리해 드리겠습니다.
