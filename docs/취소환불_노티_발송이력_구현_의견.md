# 취소/무효/환불 노티 발송 이력 – 구현 방안 의견

## 질문 요약

- 무효·환불 처리 시 가맹점에 노티를 보낸 내역을 **어디서** 볼 수 있는가?
- 발송 **성공/실패**를 알 수 있어야 함.
- 결제·무효·환불의 성공 유무를 구분해 볼 수 있어야 함.
- **방안 A**: 기존 로그 분석(피지 노티 등)에 연동해 결제/무효/환불 구분 표시  
- **방안 B**: 취소환불 메뉴 안에 **거래노티**를 두어 취소·무효·환불 관련 노티를 한곳에서 관리  

---

## 의견: 방안 B(거래노티 전용 페이지) 채택

**취소환불 > 거래노티** 로 무효/환불 노티 발송 이력을 전용 관리하는 쪽을 권장했습니다.

### 이유

1. **역할이 다름**  
   - **피지 노티**: PG → 우리 미들웨어로 **들어온** 수신 로그 (결제 성공, 취소 등).  
   - **무효/환불 노티**: 우리가 ChillPay API·동기화 등으로 처리한 뒤 **가맹점/전산에 보낸** 발송 로그.  
   수신 로그와 발송 로그를 한 화면에 섞으면 “결제/무효/환불” 구분과 “수신/발송” 구분이 같이 들어가서 복잡해집니다.

2. **담당자 업무 흐름**  
   취소·무효·환불을 다루는 담당자는 **취소환불** 메뉴에서 거래내역·무효거래·환불거래를 보면서, 같은 메뉴 안 **거래노티**에서 “이 건에 대해 노티를 보냈는지, 가맹점/전산 수신이 성공했는지”를 한 번에 확인하는 편이 자연스럽습니다.

3. **결제 성공 노티와 분리**  
   기존 피지 노티는 “결제 성공 시 PG가 보낸 노티를 우리가 가맹점에 릴레이한 이력”이 이미 있습니다.  
   무효/환불 노티는 “우리가 무효·환불 처리한 뒤 가맹점/전산에 **추가로** 보낸 노티”이므로, **거래노티**에서만 “무효/환불”로 구분해 발송 이력과 성공 유무를 보는 것이 더 명확합니다.

4. **확장**  
   나중에 “취소”에 대한 발송 이력을 넣거나, 발송 재시도·상세 사유 등을 넣어도 **거래노티** 한 곳만 확장하면 됩니다.

---

## 구현 내용 (반영됨)

- **거래노티 로그**  
  무효/환불 노티를 보낼 때마다 `data/void-refund-noti.log` 에 한 줄씩 기록.  
  - 발송일시, 구분(무효/환불), TransactionId, OrderNo, 가맹점, Route No., **가맹점 수신(성공/실패)**, **전산 수신(성공/실패/미전송)**

- **취소환불 > 거래노티** 페이지  
  - 위 로그를 최근 7/30/90일 단위로 조회.  
  - 구분 필터: 전체 / 무효 / 환불.  
  - 테이블에서 발송일시, 구분, TransactionId, OrderNo, 가맹점, Route No., 가맹점 수신, 전산 수신(성공 유무) 확인 가능.

- **취소거래** 메뉴/페이지명을 **거래내역**으로 변경.

이렇게 해서 “무효·환불 처리 시 가맹점에 노티 보낸 내역”과 “그 발송의 성공 유무”는 **취소환불 > 거래노티**에서만 보시면 됩니다.
