# 취소·무효·환불 노티 표시 위치 및 구조 정리

문의 내용에 대한 답변과, ChillPay 전화 무효 등 “미들웨어 밖에서 처리된 건”을 명확히 보기 위한 구조 정리입니다.

---

## 1. 용어 정리 (이 프로젝트 기준)

| 구분 | 설명 | ChillPay 노티 | 우리 미들웨어 역할 |
|------|------|----------------|---------------------|
| **취소 (Cancel)** | 결제 중간 고객/시스템 취소 | ✅ ChillPay가 우리에게 노티 발송 | 수신 후 가맹점/전산에 **릴레이** |
| **무효 (Void)** | 결제 완료 후 미정산 건 취소 | ❌ ChillPay 노티 없음 | API로 요청/조회 후 **우리가** 가맹점/전산에 노티 발송 |
| **환불 (Refund)** | 이미 정산된 건 취소 | ❌ ChillPay 노티 없음 | API로 요청/조회 후 **우리가** 가맹점/전산에 노티 발송 |

- **취소**는 “PG → 미들웨어 수신 → 가맹점/전산 릴레이”만 있고, **무효/환불**은 “미들웨어가 ChillPay API로 처리한 뒤 우리가 직접 노티 발송”하는 구조입니다.

---

## 2. “취소 건은 취소내역에는 있는데 거래노티에는 없다”는 이유

- **취소내역**  
  - PG에서 **수신한** 노티 중 `PaymentStatus = Cancel` 인 건만 필터한 목록입니다.  
  - 즉, **피지노티(피니결과)** 에 쌓인 로그 중 “취소”로 분류된 행을 모아서 보여주는 화면입니다.

- **거래노티**  
  - **무효/환불** 시에만, “우리가 가맹점/전산에 **보낸** 무효·환불 노티”를 기록합니다.  
  - `data/void-refund-noti.log` 에만 적재하며, **취소(Cancel)는 이 로그에 전혀 넣지 않습니다.**

그래서:

- 취소 건은 **취소내역(＝피지 수신 로그 중 취소)** 에는 있지만,  
- **취소환불 > 거래노티**에는 **의도적으로** 나오지 않습니다.  
- 취소에 대한 “노티 보낸 내역”은 아래 3번처럼 **피지노티 피니결과**와 **전산결과**에서만 확인할 수 있습니다.

---

## 3. 노티가 “어디에” 나오는지 (표시 위치)

### 3.1 취소 (Cancel) – ChillPay가 보낸 노티를 우리가 릴레이

- **피지노티 / 피니결과**  
  - ChillPay → 미들웨어 **수신** 건이 쌓이는 곳입니다.  
  - 취소 건도 여기 한 건으로 들어오고, 해당 건의 가맹점 릴레이 성공 여부(relayStatus 등)를 볼 수 있습니다.

- **전산노티 / 전산결과**  
  - 미들웨어가 전산 URL로 **보낸** 건이 쌓이는 곳입니다.  
  - 취소 건을 전산으로 릴레이했으면, 그 전송 결과가 전산결과에 한 건으로 나옵니다.

- **거래노티**  
  - **나오지 않습니다.**  
  - 거래노티는 “무효/환불 **발송** 이력” 전용이기 때문입니다.

정리: **취소 노티 발송 이력은 “피지노티 피니결과 + 전산노티 전산결과”에서만 확인 가능하고, 거래노티에는 안 나옵니다.**

---

### 3.2 API로 취소/무효/환불 “내역을 찾은 뒤” 노티를 보낸 경우

문의하신 “우리가 API를 통해서 취소내역을 찾으면 … 같은 방식으로 노티가 발송되는데, 그 노티는 어디에 나오나요?”는, 실제 구현상 **무효/환불**을 API로 “찾는” 경우에 해당합니다.

- **취소(Cancel) 건**  
  - ChillPay가 우리에게 노티를 보내서만 알 수 있고, “취소 전용 검색 API”로 목록을 조회하는 구조는 없습니다.  
  - 그래서 “API로 취소내역을 찾는다”는 표현은 **무효/환불 조회 API(Search Void / Search Refund)** 를 말하는 것으로 이해하는 것이 맞습니다.

- **무효/환불**  
  - Search Void / Search Refund로 ChillPay에서 **이미 무효/환불 처리된 건**을 조회한 뒤,  
  - 우리가 가맹점/전산에 **무효/환불 노티를 직접 보내는** 구조입니다.  
  - ChillPay는 무효/환불 시 노티를 보내지 않으므로, “같은 방식으로 노티가 발송된다”는 것은 **우리 미들웨어가 보내는 것**입니다.

이때 우리가 보낸 무효/환불 노티의 **표시 위치**는 다음과 같습니다.

- **거래노티**  
  - **여기에 나옵니다.**  
  - `sendVoidOrRefundNoti()` 호출 시마다 `appendVoidRefundNotiLog()`로 `void-refund-noti.log`에 기록하고, 취소환불 > 거래노티 화면은 이 로그를 읽어서 보여줍니다.

- **전산노티 / 전산결과**  
  - 무효/환불 노티를 전산 URL로 보낼 때 `appendInternalLog()`가 호출되므로, **전산결과**에도 “전산으로 보낸 한 건”으로 나옵니다.  
  - 다만 전산결과 화면은 “결제/취소/무효/환불” 구분 없이 전산 전송 이력만 나열하므로, “이 건이 무효/환불 노티다”는 것은 **거래노티**에서 보는 것이 더 명확합니다.

- **피지노티 피니결과**  
  - PG → 우리 **수신** 로그만 있으므로, “우리가 나중에 보낸 무효/환불 노티”는 여기에는 **별도 행으로 나오지 않습니다.**  
  - (기존 결제 성공 건이 피니결과에 있고, 그 건에 대해 나중에 무효 노티를 보냈는지는 “거래노티”나 “거래내역”의 무효/환불 노티 발송 여부 컬럼으로 확인)

**요약:**  
API로 찾은 무효/환불 건에 대해 우리가 보낸 노티는  
- **거래노티** → 발송 이력·가맹점/전산 수신 성공 여부  
- **전산결과** → 전산 전송 한 건 (타입 구분 없음)  
- **피지노티 피니결과** → 해당 무효/환불 “발송” 건은 없음 (수신 로그만 있음)

---

## 4. 무효 거래 – “발견되면 자동으로 노티” 여부

- **우리 미들웨어에서 무효 요청한 경우**  
  - **무효 요청** 버튼 → Request Void API 호출 → 성공 시 `sendVoidOrRefundNoti(log, 'void')` 호출  
  - → 가맹점 callback/result + 전산에 무효 노티 자동 발송됩니다.  
  - 이 발송 이력은 **거래노티**에 남습니다.

- **ChillPay에서 이미 무효 처리된 건(수동 무효 등)을 “발견”한 경우**  
  - **취소환불 > 무효거래** 페이지의 **“ChillPay 무효 건 동기화”** 버튼으로  
  - Search Void API로 최근 7일 무효 건을 조회한 뒤,  
  - 우리 `NOTI_LOGS`(pg 노티 로그)의 **결제 성공 건**과 `transactionId`로 매칭하고,  
  - **아직 무효 노티를 보내지 않은 건만** 가맹점/전산에 무효 노티 전송합니다.  
  - 즉, “무효가 발견되면(조회되면) 미전송 건만 자동으로 노티”하는 구조가 이미 있습니다.  
  - 단, **매칭 조건**: 해당 거래의 **결제 성공 로그**가 우리 `NOTI_LOGS`에 있어야 합니다.  
    - 우리가 결제 성공 노티를 한 번도 받지 못한 건은 매칭되지 않아, 동기화해도 무효 노티를 보내지 않습니다.

동일하게 **환불**도  
- **취소환불 > 환불거래**의 **“ChillPay 환불 건 동기화”** 로 Search Refund 후,  
- 우리 로그와 매칭된 건 중 미전송 건만 환불 노티를 보내는 구조입니다.

---

## 5. “ChillPay에서만 무효 처리된 건”이 우리 화면에 안 보이는 이유

상황:  
- ChillPay에 전화 등으로 **수동 무효** 요청 → ChillPay에는 무효로 처리됨.  
- 우리 미들웨어에서는 “무효 요청”을 하지 않았고, 노티도 우리가 보낸 적 없음.  
- 그래서 **거래노티**에는 해당 건이 없고, **무효거래** 목록도 “우리 pg 로그에 있는 성공 건 중 무효 가능 시간대”만 보여주기 때문에, “ChillPay에서만 무효된 건”은 그 목록에조차 없을 수 있습니다.

이렇게 되는 이유는 다음 두 가지입니다.

1. **무효거래 페이지**  
   - 우리가 가진 **결제 성공 로그(NOTI_LOGS)** 중에서 “무효/환불 가능 시간대”인 건만 나열합니다.  
   - ChillPay 쪽에서만 무효된 건은 “우리가 무효 요청한 내역”이 없으므로, “무효 요청” 버튼이나 “이미 무효 노티 보냄” 표시가 붙을 대상이 우리 로그에만 있습니다.  
   - 반대로, **ChillPay에는 무효로 있지만 우리 pg 로그에는 결제 성공 건이 없는 경우**는 애초에 무효거래 목록에 올라오지 않습니다.

2. **동기화(Search Void)**  
   - “ChillPay 무효 건 동기화”는 Search Void로 가져온 **무효 건**과 **NOTI_LOGS의 결제 성공 건**을 매칭합니다.  
   - 매칭되면 미전송 분만 무효 노티를 보내고, 그때부터 **거래노티**에 나타납니다.  
   - 매칭이 안 되는 건(우리 쪽에 결제 성공 로그가 없는 ChillPay 무효 건)은:  
     - 노티를 보낼 수 없고,  
     - 현재 화면에는 “ChillPay에는 무효로 있지만 우리는 모르는 건” 목록을 따로 보여주지 않습니다.

그래서 “ChillPay에서는 무효인데, 우리 노티 발송 이력/우리 미들웨어 무효 처리 이력에는 없다”는 상황이 생깁니다.

## 6. 구조를 명확히 하기 위한 제안

아래처럼 구분해 두면, “어디서 처리된 건인지”와 “노티를 우리가 보냈는지”를 헷갈리지 않게 정리할 수 있습니다.

### 6.1 노티 표시 위치 정리 (참고용 표)

| 구분 | 피지노티 (피니결과) | 전산노티 (전산결과) | 취소환불 > 거래노티 |
|------|----------------------|----------------------|----------------------|
| **취소 (Cancel)** | ✅ 수신 건 1건 (릴레이 여부 포함) | ✅ 전산 전송 1건 | ❌ 기록 안 함 |
| **무효 (Void)** – 우리가 요청/동기화 후 발송 | 해당 결제 성공 건만 존재 | ✅ 전산 전송 1건 | ✅ 무효 발송 이력 |
| **무효 (Void)** – ChillPay 전화 등 (동기화 전) | - | - | ❌ (동기화 후 매칭되면 그때부터 표시) |
| **환불 (Refund)** | 해당 결제 성공 건만 존재 | ✅ 전산 전송 1건 | ✅ 환불 발송 이력 |

### 6.2 “ChillPay에만 있는 무효/환불”을 보이게 (선택 개선)

- **무효거래** (또는 별도 블록)에서  
  - “ChillPay 무효 건 동기화” 실행 시,  
  - Search Void 결과 중 **우리 로그와 매칭되지 않은 건**을  
  - “ChillPay에는 무효로 있으나, 우리 pg 로그에 결제 성공 건이 없어 노티 미전송” 목록으로 표시해 주는 방법을 검토할 수 있습니다.  
- 이렇게 하면 “전화로만 ChillPay에 무효 처리된 건”이 ChillPay에는 있는데 우리 화면에는 없다는 상황을 줄일 수 있습니다.  
- 구현 시에는 Search Void 응답을 한 번 더 저장하거나, 동기화 시 “매칭 실패 건”만 모아서 별도 테이블/섹션으로 노출하는 방식이 필요합니다.

### 6.3 취소(Cancel) 발송 이력이 꼭 필요할 때

- 현재는 취소는 **거래노티**에 남기지 않고, **피지노티 피니결과**(수신+릴레이 여부)와 **전산결과**에서만 확인 가능합니다.  
- 운영상 “취소도 거래노티처럼 발송 이력을 한곳에서 보고 싶다”면,  
  - ChillPay에서 취소 노티를 수신해 가맹점/전산에 릴레이할 때  
  - `void-refund-noti.log`와 동일한 형식은 아니더라도, “취소 노티 발송” 전용 로그/화면을 추가하는 방안을 검토할 수 있습니다.

## 7. 요약

1. **취소 건이 취소내역에는 있는데 거래노티에는 없는 이유**  
   - 취소내역 = 피지 수신 로그 중 취소.  
   - 거래노티 = 무효/환불 **발송** 이력만 기록.  
   - 따라서 취소는 의도적으로 거래노티에 넣지 않습니다.

2. **API로 찾은 (무효/환불) 건에 대해 우리가 보낸 노티 위치**  
   - **거래노티**에만 “무효/환불 발송 이력”으로 나옵니다.  
   - 전산으로 보낸 건은 **전산결과**에도 1건씩 나오지만, “무효/환불” 구분은 거래노티에서 보는 것이 맞고,  
   - **피지노티 피니결과**에는 해당 “발송” 건은 없습니다(수신 로그만 있음).

3. **무효가 발견되면 자동으로 노티**  
   - 우리가 Request Void 한 건: 요청 성공 시 자동으로 가맹점/전산에 무효 노티 발송.  
   - ChillPay에서 이미 무효 처리된 건: “ChillPay 무효 건 동기화”로 Search Void 후, 우리 로그와 매칭된 미전송 건만 자동 발송.  
   - 둘 다 발송한 건은 **거래노티**에 기록됩니다.

4. **ChillPay 전화 무효**  
   - ChillPay에는 무효로 있지만, 우리가 요청한 게 아니면 “무효 요청” 이력이 없고,  
   - 동기화 시 **우리 pg 로그에 해당 결제 성공 건이 있어야만** 매칭되어 무효 노티 전송·거래노티 표시가 됩니다.  
   - 결제 성공 로그가 없으면 현재는 “ChillPay에만 있는 무효”를 우리 화면에서 구분해 보여주지 않으므로, 위 6.2처럼 “매칭 실패 무효 건” 목록을 노출하는 방식으로 구조를 명확히 할 수 있습니다.

이 문서는 `docs/취소환불_노티_발송이력_구현_의견.md`, `docs/ChillPay_Transaction_Services_API_분석_및_취소무효_자동화.md`와 함께 보시면 됩니다.

## 8. 무효/환불 동기화 시 NOTI_LOGS 매칭 근거 (분석)

ChillPay Search Void / Search Refund로 조회한 건과 우리 `NOTI_LOGS`(결제 성공 로그)를 **어떤 기준으로 매칭하는지** 코드 기준으로 정리합니다.

### 8.1 매칭 키: TransactionId 단일 필드

- **ChillPay 쪽 (Search Void/Refund 응답의 각 `item`)**  
  - `txId` = `item.transactionId` 또는 `item.TransactionId` (문자열로 사용)

- **NOTI_LOGS 쪽 (각 로그 `log`)**  
  - `log.body`에서 `body.TransactionId` 또는 `body.transactionId`를 꺼낸 값 → `logTxId` (문자열)

- **매칭 조건**  
  - **`logTxId === txId`**  
  - 즉, **TransactionId가 같으면 같은 거래로 간주**합니다.  
  - OrderNo, PaymentDate, Amount, MerchantCode 등은 **매칭에 사용하지 않습니다.**

### 8.2 그 외 필수 조건 (매칭 후 노티 전송 가능 여부)

TransactionId가 같다고 해서 바로 노티를 보내는 것이 아니라, 아래를 모두 만족하는 로그만 사용합니다.

| 조건 | 설명 |
|------|------|
| **결제 성공 건** | `body.PaymentStatus === 1` 또는 `'1'` 또는 `body.paymentStatus === 'Success'` 또는 `body.status === 1` |
| **가맹점 존재** | `log.merchantId`가 있고, `MERCHANTS.get(log.merchantId)`가 존재(유효한 가맹점) |

- 따라서 NOTI_LOGS에는 “결제 성공”으로 들어온 건만 있고, 그 건의 `body`에 **TransactionId**가 들어 있어야 ChillPay 무효/환불 건과 매칭됩니다.
- 결제 성공 노티를 한 번도 받지 못한 거래(로그 자체가 없음)나, 로그에 TransactionId가 없는 경우에는 매칭되지 않습니다.

### 8.3 코드 위치 (참고)

- **무효 동기화**  
  - `server.js` 내 `syncChillPayVoidNoti()`  
  - ChillPay Search Void 결과 `list`를 순회하면서, 각 `item`의 `transactionId`/`TransactionId`로 `NOTI_LOGS.find(...)` 호출.  
  - find 콜백: `logTxId === txId` && 결제 성공 && 가맹점 유효.
- **환불 동기화**  
  - `syncChillPayRefundNoti()`  
  - 동일한 방식으로 `NOTI_LOGS.find(...)` 시 `logTxId === txId` 및 위 조건 사용.

### 8.4 요약

- **매칭 근거**  
  - **TransactionId 일치 한 가지**입니다.  
  - ChillPay 무효/환불 건의 `transactionId`와 NOTI_LOGS 항목의 `body.TransactionId`(또는 `transactionId`)가 같으면 “같은 거래”로 보고, 해당 로그로 가맹점/전산에 무효(또는 환불) 노티를 보냅니다.
- **매칭 실패가 나는 경우**  
  - ChillPay에는 무효/환불로 있지만, 우리 NOTI_LOGS에 해당 TransactionId를 가진 **결제 성공** 로그가 없을 때  
  - 또는 해당 로그에 merchantId가 없거나, 가맹점이 삭제/비활성화된 경우  
  → 노티를 보내지 않고, 거래노티에도 올라오지 않습니다.
